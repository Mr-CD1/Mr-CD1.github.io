dig简介

Dig是在类UNIX命令行模式下查询DNS包括NS记录、A记录、MX记录等相关信息的工具

输出组成

示例代码：
[root@Gin scripts]# dig www.cnblogs.com

; <<>> DiG 9.8.2rc1-RedHat-9.8.2-0.30.rc1.el6 <<>> www.cnblogs.com
;; global options: +cmd  
//Dig的部分输出告诉我们一些有关于它的版本信息(version 9.2.3)和全局的设置选项，如果+nocmd在命令行下是第一个参数的话，那么这部分输出可以通过加+nocmd的方式查询出来

;; Got answer:
;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 41440
;; flags: qr rd ra; QUERY: 1, ANSWER: 1, AUTHORITY: 0, ADDITIONAL: 0 
//在这里，Dig告诉我们一些从DNS返回的技术信息，这段信息可以用选项 +no]comments来控制显示，但是小心，禁止掉comments也可能关闭一些其它的选项。

;; QUESTION SECTION:
;www.cnblogs.com.               IN      A 
//在这个查询段中，Dig显示出我们查询的输出，默认的查询是查询A记录，你可以显示或者禁止掉这些用+[no]question选项

;; ANSWER SECTION:
www.cnblogs.com.        349     IN      A       42.121.252.58 
//最后，我们得到我们查询的结果。www.isc.org 的地址是204.152.184.8，我不知道为什么你们更喜欢过滤掉这些输出，但是你可以用+[no]answer保留这些选项。

;; Query time: 149 msec
;; SERVER: 202.106.0.20#53(202.106.0.20)
;; WHEN: Sat Feb  4 15:26:43 2017
;; MSG SIZE  rcvd: 49 
//最后一段默认输出包含了查询的统计数据，可以用+[no]stats保留。

组成：

  1. Got answer:从DNS返回的技术信息
  2. Question SECTION:显示我们查询的输出，A是默认的查询记录类型
  3. ANSWER SECTION: 查询结果信息
  4. 查询统计数据

DNS记录类型

发表于 2019-11-12

A记录

A（Address）记录是用来指定主机名（或域名）对应IP地址记录的。即服务器的IP，域名绑定A记录就是告诉DNS，当输入域名的时候就要引导到该A记录对应的服务器上

CNAME记录

（Canonical Name）记录，也被称为规范名字（别名解析）。允许将多个名字映射到同一台计算机上。
例如：当需要将多个域名指向同一个服务器IP时，首先，可以将一个域名作为A记录指向服务器IP，然后对于其他域名就采用CNAME记录指向到A记录的域名上。
优点：当你的服务器IP更改时，你就可以不必一个个更改域名指向了，只需更改A记录的那个域名的指向即可。

MX 记录

（MX record）邮件交换记录：用于指定负责处理发送给收件人域名的邮件服务器。

NS 记录

（Name server）：用于指定该域名由哪个DNS服务器来进行解析。当你注册域名时，总会有默认的DNS服务器来对你的域名进行解析。

TXT记录

一般指为某个主机名或域名设置的说明，即DNS想让你看到的文本

正则回溯

发表于 2019-11-12

正则表达式引擎

第一类：DFA（确定型有穷自动机）

DFA对应的是文本主导的匹配。从匹配的文本开始，从左到右，每个字符不会匹配两次，速度更快，但功能有限
例：

text='after tonight'  regex='to(nite|nighta|night)'

从a开始匹配，直到第一个t跟正则的t匹配，但是e和o匹配失败，继续直到第二个t，且o与o匹配，继续n的匹配，有三个表达式，同时进行匹配直到最后

第二类：NFA（不确定型有穷自动机）

NFA则是从正则表达式下手，不断读入字符，尝试是否符合当前的正则，不匹配则重新尝试速度更慢，但功能也更为丰富
例：

text='after tonight'  regex='to(nite|nighta|night)'

从t开始匹配a，失败继续，直到文本中出现第一个t，接着匹配o和e，失败，正则回退到t继续匹配，如此循环，直到全部匹配

回溯

回溯就是使用的上述中NFA正则表达式引擎
例如：

ab{1,3}c

对于文本“abbbc”，是没有发生回溯的，检测完a后，马上匹配三个b，并不会发生回溯；
但当输入文本为“abc”时，就会发生回溯：

从第三步开始，正则发现自己没有检测到c时，正则会继续匹配b直到最大的匹配数目，最后再去匹配c（贪婪模式：尽可能多的去匹配前导字符）

如何避免？

使用懒惰模式

什么是懒惰模式？
即在该模式下，正则引擎会尽可能少的重复匹配字符，匹配成功后它会继续匹配剩余的字符串，若将上例中的正则表达式改为：

ab{1,3}?c

则匹配过程为：

第二步与b匹配之后，接着就用c和文本中的c去匹配，未发生回溯

其他正则模式

独占模式

正则表达式会尽可能长的去匹配字符串，一旦不匹配成功就会结束匹配而不发生回溯
例：

ab{1,3}+bc

在第四步中，正则中的b为能和c进行匹配，这时就结束匹配，整个文本就无法和正则表达式发生匹配了。

SQL约束攻击

发表于 2019-11-11

SQL约束攻击

攻击原理

1.在SQL中执行字符串处理时，字符串末尾的空格将被删除，例如：

SELECT * FROM users where username='vampire          ';

与执行vampire搜索语句时一样的结果（除了LIKE语句）

2.现在假设一个存在漏洞的网站。为了侵入任意用户的账户（假设用户名为admin），只需要使用“admin[空格]1”加上随机的密码注册即可。这时执行INSERT语句，它会只插入指定字符串长度的字符，后面的会被删除

3.当使用admin登录时，执行SELECT语句，返回第一个数据记录，也就是原始的数据记录。这样攻击者就可以以原始用户身份登录

防御手段

将要求或者预期具有唯一性的那些列加上UNIQUE约束（唯一性），这样就不能插入两个相同的字符串，INSERT查询将会失败

base64-隐写

发表于 2019-11-07

base64 基本原理

通过64个字符来表示二进制数据的方法，包含的字符有：a-z \A-Z \0-9 + /
还包括一种填充字符“=”

规则

1.把三个字节变成4个字节
2.每76个字节加一个换行符
3.最后的结束符也要处理

转换表

编码

字符串“Tr0”编码成“VHIw”：

每个字节8位，经过base64每个字节变成6位。上面刚好可以精确的分成四个分块。
但若字节数不是3的整数倍，那就不能刚好分块6位的块，只能在字节后添加0来使其字节数变成6的倍数
然后，在编码后的字符串后面添加1个或2个0来表示所添加的0
比如，字符串“Tr0y”经过base64编码后变成“VHIweQ==”:

解码

1.将等号起掉，转为二进制数
2.从左到右，8个一组，多于位丢掉，转为相应的ASCII码

隐写原理


在解码的时候会将多余位丢掉，所以这里的值并不会影响到解码，我们可以在这里进行隐写。
一行base64最多有2个等号，也就最多有4个可隐写位。把需要隐藏的信息转为8进制位塞进去就可以了

加密代码

 -*- coding: cp936 -*-
import base64
flag = 'Tr0y{Base64isF4n}' #flag
bin_str = ''.join([bin(ord(c)).replace('0b', '').zfill(8) for c in flag]) #将flag转换成2进制字符串
base64chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/'
with open('0.txt', 'rb') as f0, open('1.txt', 'wb') as f1: #'0.txt'是明文, '1.txt'用于存放隐写后的 base64
    for line in f0.readlines():
        rowstr = base64.b64encode(line.replace('\n', '')) #将明文用base64编码并将换行符去掉
        equalnum = rowstr.count('=') # ‘=’的数量
        if equalnum and len(bin_str):
            offset = int('0b'+bin_str[:equalnum * 2], 2) #equalnum * 2：可以隐写的位数
            char = rowstr[len(rowstr) - equalnum - 1] #等号前一位字符
            rowstr = rowstr.replace(char, base64chars[base64chars.index(char) + offset]) #
            bin_str = bin_str[equalnum*2:] #丢弃前面已隐写的位
        f1.write(rowstr + '\n')

解密代码

 -*- coding: cp936 -*-
b64chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/'
with open('1.txt', 'rb') as f:
    bin_str = '' #空字符串
    for line in f.readlines():
        stegb64 = ''.join(line.split()) #以换行符分隔base64全文
        rowb64 =  ''.join(stegb64.decode('base64').encode('base64').split()) 将其进行base64解码再进行编码
        offset = abs(b64chars.index(stegb64.replace('=','')[-1])-b64chars.index(rowb64.replace('=','')[-1])) #将base64编码中的=去掉，并得到最后一个字符偏移
        equalnum = stegb64.count('=') #no equalnum no offset
        if equalnum:
            bin_str += bin(offset)[2:].zfill(equalnum * 2)
        print ''.join([chr(int(bin_str[i:i + 8], 2)) for i in xrange(0, len(bin_str), 8)]) #8 位一组

透明代理、正向代理、反向代理

发表于 2019-10-30

1.透明代理

定义：透明代理的意思是客户端根本不需要知道有代理服务器的存在，它改变你的request fields（报文），并会传送真实IP，多用于路由器的NAT转发中

应用：防火墙
优点：防火墙本身也起到路由的作用，为用户安装防火墙的时候，需要考虑路由表。但使用透明代理，即采用无IP方式运行，用户就不必重新设定和修改路由器，防火墙可以直接安装和放置在网络中使用。类似于网桥的存在

工作示意图：

2.正向代理（客户端）

定义：相当于一个在客户端和服务器之间的中介。客户端向代理服务器发送一个请求并指定目标，然后代理服务器向原始服务器发送请求并将得到的内容返回给客户端。

例子：国内的用户向使用谷歌会被阻拦，所以这个时候把请求发送到另一个代理服务器上，由其代为转发请求和接收响应内容

3.反向代理（服务器端）

定义：类似于正向代理，它们的区别在于反向代理在服务器端，对于客户端来说，它们就是目标服务器。

通常，反向代理用作web加速，即使用反向代理作为web服务器的前置机来降低网络和服务器的负载，提高访问效率
